# AI UI Asset Tool 产品规划

## 1. 文档目标

本文档用于规划当前 Figma 插件的下一阶段能力演进，重点围绕以下 3 个目标：

1. 对 AI 生成的界面图进行元素拆解，输出透明底素材，并保留元素原始位置。
2. 将图片、icon 尽可能转为可编辑 SVG 文件。
3. 所有核心能力优先围绕 Figma 插件内运行、资产导出和 Figma 画布回填展开，暂不考虑 Sketch 输出。

这不是一份“理想状态”文档，而是一份面向实际落地的规划文档。会明确：

- 哪些能力适合先做
- 哪些能力存在模型或技术边界
- 哪些能力可以稳定交付
- 哪些能力需要分阶段推进

同时，界面体验是本工具的核心竞争力之一。后续实现时，不能只把功能堆出来，而要保证插件看起来像一个成熟的 Figma 生产力工具：

- 信息密度高但不拥挤
- 图片预览清晰、稳定、不变形
- 切图选择要直观、轻量、可回退
- 资产列表要有专业的管理感
- 状态反馈要清楚，不让用户猜当前发生了什么
- 视觉上保持克制、干净、现代，不做花哨装饰

---

## 2. 当前工具状态

当前工具已经具备的基础能力：

- 文生图 / 图生图
- 本地代理对接 GPT image 模型
- 在插件内预览生成结果
- 下载结果图
- 将整张结果图放入 Figma

当前工具还不具备的能力：

- 自动识别并拆分图片中的独立元素
- 将结果图结构化为可编辑图层
- 输出透明底元素资产包
- 输出 SVG 编辑文件
- 输出更高质量的 Figma 可编辑结果

因此，下一阶段的核心问题已经从“生成一张图”转成：

> 如何把一张 AI 生成的位图，尽可能重建成可拆解、可编辑、可复用的设计资产。

---

## 3. 核心能力定义

### 3.1 元素拆解与透明底输出

目标定义：

- 将生成结果图中的界面元素拆出来
- 每个元素导出为独立 PNG
- PNG 背景透明
- 同时保留该元素在原始画布中的位置信息

期望产物：

- `screen.png`：原始整屏图
- `elements/*.png`：拆出的透明底元素
- `manifest.json`：每个元素的坐标、尺寸、层级、类型、文件路径

典型元素类型：

- icon
- 按钮
- 卡片
- banner
- 弹窗主体
- 人物 / 商品主图
- 装饰插画
- 底部导航图标

这里需要明确一个事实：

> AI 生成结果本质上是一张扁平位图，不是真图层文件。

所以“拆图”不是直接读取图层，而是依赖：

- 视觉检测
- 分割
- 规则化裁切
- 局部抠图

这意味着拆解能力可以做，但不能假设 100% 完美，尤其在元素遮挡、融合阴影、复杂背景、渐变叠加时会有误差。

---

### 3.2 SVG 可编辑输出

目标定义：

- 对图片中的 icon、简单图形、局部插画，尽可能生成可编辑 SVG
- 输出的 SVG 应可在 Figma / Sketch / 浏览器中继续编辑

需要区分两类内容：

#### A. 适合转 SVG 的内容

- 线性 icon
- 纯色图标
- 几何图形
- 边框、按钮轮廓
- 简单插画形状

这类内容可以走：

- 边缘检测
- 矢量化
- SVG path 重建

#### B. 不适合强转 SVG 的内容

- 人物照片
- 商品渲染图
- 复杂渐变
- 光影丰富的位图插画
- 大面积纹理背景

这类内容即使转成 SVG，也往往只是“嵌入位图”或输出大量 path，编辑价值很低，文件还会非常重。

因此产品上更合理的定位不是：

> 一键把整张图全转成高质量可编辑 SVG

而是：

> 针对可矢量化的元素输出 SVG，不适合的部分继续保留为 PNG/WebP。

---

### 3.3 Figma 优先，不考虑 Sketch

当前阶段产品目标统一为：

- 在 Figma 插件内完成生成、分析、切图、导出与回填
- 在 Figma 画布中尽可能重建为可编辑节点
- 优先把 Figma 内的完整工作流闭环做稳

因此这里明确收口：

> 当前版本不把 Sketch 文件输出纳入主线规划。

原因：

- 最终宿主已经明确是 Figma
- Sketch 会额外引入文件格式、兼容性和维护成本
- 当前更重要的是把元素分析、透明切图、SVG 和 Figma 回填质量做好

后续如果未来有明确的 Sketch 交付需求，再单独作为扩展能力评估。

---

## 4. 总体技术路线

建议整体采用“四层架构”：

### 第一层：图像生成层

负责：

- 文生图 / 图生图请求
- 比例、数量、风格控制
- 返回整图结果

现状：

- 已基本具备

---

### 第二层：结构分析层

负责：

- 检测界面中的元素区域
- 判断元素类型
- 生成元素清单
- 输出布局坐标

典型输出：

```json
{
  "screen": { "width": 1024, "height": 1536 },
  "elements": [
    {
      "id": "banner_1",
      "type": "banner",
      "bbox": { "x": 48, "y": 112, "width": 928, "height": 320 },
      "zIndex": 1
    }
  ]
}
```

这是后面所有能力的基础。

没有这一层，透明切图、SVG、Figma 内结构化重建都只能靠拍脑袋。

---

### 第三层：资产重建层

负责：

- 裁切单个元素
- 去背景 / 抠图
- 输出透明 PNG
- 对适合矢量化的内容输出 SVG

这是把“检测结果”变成“可交付资产”的关键层。

---

### 第四层：Figma 输出层

负责：

- 放入 Figma
- 导出资产包
- 在 Figma 中重建结构化节点

也就是说：

- Figma 是当前主目标
- 当前不规划 Sketch 输出

---

## 5. 各目标的可行性判断

### 5.1 目标一：拆元素 + 透明底 + 保留位置

可行性：**高**

原因：

- 这是最符合当前产品方向的核心能力
- 可以先做“半自动版本”
- 能最快形成实际业务价值

建议落地方式：

#### V1

- 先支持模块级切图
- 用户通过预设或框选确认切图区域
- 导出透明 PNG 和坐标信息

#### V2

- 支持自动检测常见 UI 元素
- 输出 `asset_manifest`
- 一键将透明元素按原坐标放回 Figma

#### V3

- 支持复杂元素抠图
- 支持层级和遮挡处理

---

### 5.2 目标二：生成 SVG 可编辑文件

可行性：**中**

原因：

- 对 icon 和简单图形是可行的
- 对整张图片或复杂 UI 截图，不适合承诺“全部可编辑”

建议落地方式：

#### V1

- 仅针对 icon、按钮轮廓、简单图形输出 SVG

#### V2

- 补充插画/图标的局部矢量化

#### 不建议承诺

- 整张 UI 一键转为高质量可编辑 SVG

这条在产品表述上一定要谨慎，不然很容易形成错误预期。

---

### 5.3 当前阶段不规划 Sketch 输出

可行性判断：

- 不是不能做
- 而是当前没有必要优先做

建议定位：

> Sketch 输出从当前路线中移出，后续仅在出现明确业务需求时单独评估。

当前更合理的路线是：

1. 先做结构化 manifest
2. 再做 Figma 回填
3. 再做 Figma 内可编辑重建

---

## 6. 推荐产品路线

建议按照下面顺序推进。

### Phase 1：可用的“切图工具”

目标：

- 让生成图可以被拆成多个资产
- 导出透明 PNG
- 保留坐标

范围：

- 支持模块级切图
- 支持手动框选和预设切图模板
- 生成 `manifest.json`
- 一键回填 Figma

产出：

- 原图
- 切图元素
- 透明 PNG
- 元素坐标

这是最值得最先做的一期。

---

### Phase 2：半自动元素分析

目标：

- 自动识别常见 UI 结构

范围：

- 卡片
- 按钮
- banner
- icon 区块
- 弹窗主体
- 导航栏

产出：

- 元素清单
- 元素类型
- 布局关系

这一期做完，工具就从“切图器”升级成“结构分析器”。

---

### Phase 3：SVG 资产输出

目标：

- 将适合的元素输出成可编辑 SVG

范围：

- icon
- simple shapes
- line-based assets
- 局部可矢量化插图

不纳入本期：

- 全屏复杂 UI 位图的完整 SVG 重建

---

### Phase 4：Figma 内结构化重建增强

目标：

- 在 Figma 内把结果重建得更像真正可编辑设计稿

范围：

- 画板
- 图层分组
- 元素位置
- PNG / SVG 资产引用
- 文本与基础组件重建

本期本质上是“Figma 设计稿重建增强”，而不是导出其他设计工具文件。

---

## 7. 推荐输出数据结构

建议从现在开始统一规划一个中间层 `design manifest`，后面 Figma、SVG 和 Figma 内结构化重建都走这套中间结构。

示例：

```json
{
  "version": "1.0.0",
  "screen": {
    "name": "ecommerce_home",
    "width": 1024,
    "height": 1536,
    "targetRatio": "9:16"
  },
  "sourceImage": {
    "format": "png",
    "path": "outputs/source.png"
  },
  "elements": [
    {
      "id": "banner_main",
      "name": "Main Banner",
      "type": "banner",
      "bbox": { "x": 56, "y": 118, "width": 912, "height": 326 },
      "zIndex": 1,
      "asset": {
        "png": "outputs/elements/banner_main.png",
        "svg": null
      },
      "editable": false,
      "transparent": false
    },
    {
      "id": "icon_cart",
      "name": "Cart Icon",
      "type": "icon",
      "bbox": { "x": 884, "y": 86, "width": 48, "height": 48 },
      "zIndex": 8,
      "asset": {
        "png": "outputs/elements/icon_cart.png",
        "svg": "outputs/elements/icon_cart.svg"
      },
      "editable": true,
      "transparent": true
    }
  ]
}
```

这层结构的价值很高：

- Figma 放图靠它
- 下载资产包靠它
- SVG 输出靠它
- Figma 内结构化重建也靠它

## 8. 关于接入 DESIGN.md / awesome-design-md 的可行性

你补充的这个方向是有价值的，但当前不作为近期主线推进。

本节只作为后期规划保留：

> `DESIGN.md` 可以在后期作为设计规则层，用来增强 Figma 重建设计稿的风格一致性，但不进入当前 P0/P1/P2 开发范围。

参考仓库：

- `VoltAgent/awesome-design-md`
- 核心思路来自 Google Stitch 的 `DESIGN.md`

它的本质不是“切图引擎”，而是：

> 用一份给 AI 可读的设计规范文档，约束生成和重建出来的界面风格、组件规则、排版方式和设计语言。

也就是说，它适合解决的是：

- 生成结果风格不稳定
- 重建设计稿时组件长相不统一
- Figma 回填后看起来不像一个完整设计系统

它不直接解决的是：

- 从位图中识别图层
- 自动抠出透明元素
- 精确恢复每个元素的真实原始结构

### 8.1 这条路线能做什么

如果我们把 `DESIGN.md` 接进来，它最适合做两件事：

#### A. 作为生成阶段的风格约束层

在文生图 / 图生图之前，把 `DESIGN.md` 的关键信息注入 prompt 或系统规则中。

例如：

- 颜色体系
- 字体层级
- 按钮圆角
- 卡片阴影
- 布局密度
- 导航形态
- banner 风格

这样生成出来的图，风格会更统一，也更接近你想模仿的站点或品牌体系。

#### B. 作为“重建设计稿”的设计规则层

这件事更重要。

如果我们未来要实现：

- 生成图片后
- 在 Figma 里自动重建一个可编辑设计稿

那单靠图片分析是不够的。因为图片只能告诉我们：

- 这里像一个按钮
- 那里像一个卡片
- 这里有一个 icon

但它不能告诉我们：

- 按钮应该用什么圆角半径
- 主标题用什么字号和行高
- 卡片边框该不该有阴影
- 间距体系是不是 8pt grid

而 `DESIGN.md` 正好可以补这层“设计规范语义”。

所以更合理的流程不是：

> 生成图片 -> 直接从位图还原 Figma 图层

而是：

> 生成图片 -> 做结构分析 -> 结合 DESIGN.md 规则 -> 在 Figma 中重建设计稿

这条链路是成立的，而且比直接从位图暴力还原更靠谱。

---

### 8.2 这条路线不能解决什么

这里也要讲清楚，不然很容易对它期待过高。

`DESIGN.md` 不能直接做到：

- 把整张生成图一键变成完美可编辑 Figma 稿
- 自动还原真实图层关系
- 自动恢复每个元素的原始矢量路径
- 自动从位图中抠出透明底素材

原因很简单：

`DESIGN.md` 是“规则描述”，不是“图像理解模型”。

它更像：

- 一份设计系统说明书
- 一份 UI 风格约束文档
- 一份 AI 在重建界面时的参考标准

所以它能提升：

- 风格一致性
- 组件合理性
- 设计稿重建质量

但不能替代：

- 元素检测
- 图像分割
- 抠图
- 矢量化

---

### 8.3 后期最合理的接入方式

我建议把 `DESIGN.md` 放在后期的“设计稿重建增强层”，而不是放在“切图层”，也不放在当前第一阶段。

推荐接入方式如下：

#### 方式 1：预设设计规范模板

插件内支持选择：

- 默认设计规范
- 某个品牌/网站风格模板
- 用户自定义 `DESIGN.md`

作用：

- 约束生成阶段
- 约束 Figma 重建阶段

这是后期最容易落地的一版，但不建议现在优先做。

#### 方式 2：重建 Figma 设计稿时读取 DESIGN.md

流程：

1. 生成整图
2. 分析布局结构
3. 判断哪些区域是按钮、卡片、文本、图标、导航
4. 用 `DESIGN.md` 中的规则，去创建对应 Figma 节点

例如：

- 按钮用哪种圆角
- 文本用哪种字重
- 卡片用哪种阴影
- 间距遵守哪套 spacing scale

这一版是最符合你“自动生成设计稿”目标的。

#### 方式 3：把 DESIGN.md 作为中间层导出的一部分

未来如果要做：

- Figma 可编辑稿
- SVG 设计规范输出

都可以让 `design manifest + DESIGN.md` 一起作为中间输入。

也就是说：

- `manifest` 负责结构和坐标
- `DESIGN.md` 负责视觉规范和组件规则

这两层结合，才有可能把 AI 生成图变成“结构化设计文件”。

---

### 8.4 可行性与优先级判断

我的判断是：

**可行，值得保留在规划中，但优先级靠后。**

优先级建议：

- 不是拿来替代“切图”
- 是拿来增强“自动重建设计稿”

所以它更适合放在：

- Phase 4 之后
- 或作为 Figma 内可编辑重建稳定后的增强能力

原因：

在没有元素结构分析能力之前，单独接 `DESIGN.md` 只能影响“生成风格”，但还不能真正稳定地产出可编辑设计稿。

而一旦我们有了：

- 元素清单
- 坐标
- 类型识别

再接 `DESIGN.md`，就能明显提升：

- Figma 重建质量
- 设计系统一致性
- 组件规范一致性

---

### 8.5 产品结论

这一点我建议正式写进路线里：

> 本工具后期可以接入 `DESIGN.md` 体系，作为 AI 设计规则层，用于约束生成风格，并在 Figma 重建设计稿时提供统一设计系统规则。

更准确地说，它适合支持的是：

- “生成图像后，按照某个设计规范在 Figma 中重建设计稿”

而不是：

- “生成图像后，直接靠 DESIGN.md 自动拆图”

前者是合理路线，后者不是它的能力边界。

当前阶段的产品判断：

- 先不做 `DESIGN.md` 接入
- 先把切图、透明 PNG、SVG、Figma 回填做稳
- 等 Figma 内结构化重建进入稳定阶段后，再评估接入 `DESIGN.md`

---

## 9. 风险与边界

### 9.1 位图天生没有图层语义

风险：

- 模型输出只有像素，没有原始图层

影响：

- 元素拆分和可编辑输出都只能靠“推断”

应对：

- 把产品表述从“还原图层”改成“结构化重建”

---

### 9.2 透明底不等于简单裁切

风险：

- 背景复杂时，单纯裁切不能得到透明资产

影响：

- 需要分割或抠图

应对：

- 把透明 PNG 作为高级能力分阶段推进

---

### 9.3 SVG 可编辑存在质量边界

风险：

- 位图强转 SVG 可能不可维护

应对：

- 只对适合矢量化的内容启用 SVG 输出

---

### 9.4 Figma 内可编辑重建依赖前置结构能力

风险：

- 如果前面的元素识别不稳定，那么回填到 Figma 的也只能是“位图 + 少量资产”

应对：

- 优先把元素识别、透明切图、SVG 输出做稳，再提升 Figma 内结构化重建质量

---

## 10. 产品建议结论

基于当前目标，我建议我们把这三个能力的优先级定成：

### P0

**切图 + 透明 PNG + 保留位置**

这是最核心、最现实、最容易形成产品闭环的一条线。

### P1

**SVG 输出（仅针对 icon / 简单图形）**

可以做，而且很有价值，但必须控制承诺范围。

### P2

**Figma 内可编辑重建增强**

包括：

- 更细的图层结构
- 更好的文本恢复
- 更稳定的 SVG / PNG 回填

### P3

**DESIGN.md 设计规则层**

后期再评估，用于提升 Figma 重建后的风格一致性和设计系统规范性。

## 11. 建议的下一步

建议下一步先出第二份更具体的执行文档，把 Phase 1 拆成可以开发的模块。

建议拆法：

1. 切图数据结构设计
2. 元素检测策略
3. 透明 PNG 导出策略
4. Figma 回填策略
5. 手动框选与自动检测的交互设计
6. Figma 内结构化回填策略

如果按这个顺序推进，我们很快就能得到一个真正可用的 V1：

> 生成 UI 图 -> 识别模块 -> 导出透明 PNG -> 带坐标回填 Figma

这会比直接硬上 SVG 或其他设计文件输出更稳，也更符合你这个工具当前的阶段。

## 12. 一句话版本

这个工具后续最合理的路线不是：

> 一步把 AI 位图直接变成完整可编辑设计稿

而是：

> 先把整图变成“有结构的资产清单”，再逐步推进透明切图、SVG 和 Figma 内可编辑重建。
