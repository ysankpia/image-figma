# AI UI Asset Generator 2.0 Roadmap

版本：草案 1

最后更新：2026-05-13

本文档用于记录 AI UI Asset Generator 2.0 的功能规划、优先级、技术实现方案和后续变更。后续如果功能范围有调整，应同步更新本文档，避免需求散落在聊天记录里。

## 1. 2.0 版本目标

1.0 版本已经跑通了基础流程：

- 文生图 / 图生图
- 多供应商配置
- 设计稿预览
- 手动切图
- 透明 PNG 资产
- 放入 Figma

2.0 的目标是把工具从“能生成和切图”升级为“更像专业 Figma 生产力插件”：

> 支持可调整窗口、配置持久化、多设计稿选择、可编辑切图区域、资产管理、SVG 实验转换和更清晰的 Figma 回填。

核心方向不是堆更多模型，而是让 AI 生成图可以更稳定地进入 Figma 工作流，并变成可管理、可复用的设计资产。

补充实验方向：

- [可编辑设计稿导入实验方案](./editable-design-import-experiment.md)：在 `放入 Figma` 时保留 `源文件`，并新增 `编辑设计稿`，后者尝试把生成图片还原成 HTML 预览，再导入为可编辑 Figma 图层。该能力必须保持实验入口，不影响当前源文件导入。
- ScreenCoder 思路：先独立验证 `图片 -> 编辑设计稿预览` 的还原质量，再决定是否进入 Figma 图层转换。
- web-to-figma 思路：已接入 H5 DOM 捕获桥，把预览 iframe 中的文字、图片、基础形状、圆角、边框、阴影和基础渐变映射成 Figma 节点；它是 `HTML -> Figma` 的桥，不负责图片理解。后续继续补齐复杂背景、mask、filter、伪元素等高级 CSS。

## 2. 2.0 开发原则

2.0 必须在保证 1.0 功能完整可用的基础上逐步增强。任何新功能都不能破坏已经跑通的 1.0 主流程。

### 2.1 兼容 1.0

以下 1.0 能力必须持续可用：

- 文生图
- 图生图
- 多供应商配置
- 生成结果预览
- 多图轮播
- 手动切图
- 透明 PNG 资产
- 下载生成图
- 放入 Figma
- 本地预览环境
- Figma 插件环境

### 2.2 渐进增强

2.0 新功能优先以渐进方式接入：

- 不直接替换 1.0 主流程
- 先作为增强能力或实验入口存在
- 稳定后再提升为默认体验
- 新旧流程并存一段时间，方便对比和回退

### 2.3 可回退

关键能力必须保留回退路径：

- 切图区域编辑失败时，可以继续使用 1.0 手动切图
- SVG 转换失败时，保留原 PNG 资产
- 背景修复失败时，保留原始设计稿
- Figma 回填失败时，不影响本地生成结果和切图资产
- 配置读取失败时，用户可以重新手动填写
- 新布局异常时，应能恢复默认窗口尺寸

### 2.4 分阶段提交

2.0 开发期间每一轮功能应独立提交，方便定位问题和回退。

推荐提交粒度：

1. 窗口 resize + 迷你模式
2. API 配置持久化
3. 多设计稿选择流程
4. 切图区域可编辑
5. 资产面板升级
6. Figma 回填优化
7. 背景修复和 SVG 实验能力

### 2.5 GitHub 推送策略

2.0 开发期间不自动推送到 GitHub。开发过程可以在本地提交，等 2.0 功能完成并验证后，再统一推送到远程仓库。

### 2.6 每轮验证清单

每一轮开发完成后至少验证：

- 文生图生成 1 张
- 文生图生成多张
- 图生图生成
- 生成结果轮播
- 下载生成图
- 手动切图
- 透明 PNG
- 放入 Figma
- 配置保存和读取
- Figma 插件环境可运行
- 本地模拟环境可运行

## 3. 2.0 P0 范围

P0 是 2.0 必须优先解决的能力。只有 P0 稳定后，再考虑 P1/P2。

### 3.1 插件窗口可调整大小

#### 目标

插件窗口不再依赖固定尺寸。用户可以根据自己的屏幕大小调整窗口，插件内部布局自动适配。

#### 需求

- 支持 Figma 插件窗口 resize
- 记住用户上次窗口尺寸
- 大屏保持左右布局
- 小屏时内容可滚动，核心操作按钮不能消失
- 生成图预览、切图区域、资产面板都需要适应容器变化

#### 技术方案

- `code.js` 启动时从 `figma.clientStorage` 读取窗口尺寸
- 使用 `figma.showUI(html, { width, height })` 设置默认窗口
- 前端窗口尺寸变化后通知主线程保存
- UI 使用 CSS grid/flex 组织布局
- 图片预览层使用容器尺寸计算缩放比例
- 切图坐标始终基于原图真实尺寸，而不是屏幕显示尺寸

#### 验收标准

- 大屏、小屏下插件都可正常操作
- 生成按钮、设置按钮、切图按钮不会因为窗口变小不可见
- 切图坐标不因窗口尺寸变化而错位

---

### 3.2 插件窗口折叠 / 迷你模式

#### 目标

Figma 插件不支持真正的系统级最小化，但可以通过 resize 实现“类最小化”的折叠模式。

#### 需求

- 顶部提供收起按钮
- 收起后插件变成迷你窗口
- 迷你窗口展示插件名称、当前状态和展开按钮
- 展开后恢复到上次正常窗口尺寸
- 记住上次折叠状态

#### 技术方案

- 使用 `figma.ui.resize(width, height)` 切换窗口尺寸
- 正常窗口尺寸保存为 `lastWindowSize`
- 迷你模式尺寸建议先定为 `320 x 72`
- 不优先使用 `figma.ui.hide()`，避免用户不知道如何恢复窗口

#### 验收标准

- 点击收起后窗口能变小
- 点击展开后恢复原尺寸
- 生成中、切图中等状态在迷你窗口仍可见

---

### 3.3 API 配置本地持久化

#### 目标

用户填写一次供应商配置后，下次打开插件自动读取，不需要每次重新输入 API Key。

#### 需求

- 保存供应商类型
- 保存 Base URL
- 保存模型名称
- 保存 API Key
- 支持修改配置
- 支持清空配置
- 支持测试连接
- 不把真实 key 写死在代码里
- 不把真实 key 提交到 GitHub

#### 技术方案

- Figma 环境优先使用 `figma.clientStorage`
- 浏览器模拟环境使用 `localStorage`
- 后端 `.env` 只作为兜底配置
- `.env`、`.local-provider-config.json` 保持在 `.gitignore`
- 设置面板初始化时先读取本地配置，再渲染输入框

#### 验收标准

- 第一次填写保存后，关闭再打开仍能读取
- 切换供应商后保存生效
- 清空配置后不会继续使用旧 key
- Git 仓库中不出现真实 API Key

---

### 3.4 多设计稿选择流程

#### 目标

生成多张设计稿时，用户先选择其中一张作为当前设计稿，再进入切图模式。

#### 需求

- 生成结果支持多张
- 支持轮播切换
- 支持缩略图选择
- 当前选中图有明确状态
- 切图资产绑定到对应设计稿
- 切换设计稿时不混用其他图片的切图资产

#### 技术方案

- 前端维护 `generatedImages[]`
- 使用 `activeImageIndex` 表示当前设计稿
- 每张图生成唯一 `imageId`
- 切图数据结构按 `imageId` 关联
- 进入切图模式时只操作当前选中设计稿

#### 验收标准

- 生成 2/4 张图时能稳定切换
- 选中第一张切图后，切到第二张不会显示第一张资产
- 放入 Figma 时只导入当前选中的设计稿和资产

---

### 3.5 切图区域可编辑

#### 目标

切图框不再是一次性结果。用户切完后可以继续调整大小、位置和名称。

#### 需求

- 切图框可拖拽移动
- 切图框可拖拽边角缩放
- 支持删除切图
- 支持重命名切图
- 支持调整后重新生成透明 PNG
- 支持键盘微调可作为 P1，但数据结构需要预留

#### 技术方案

- 每个切图区域保存真实原图坐标：

```json
{
  "id": "slice_01",
  "imageId": "image_001",
  "name": "icon_calendar",
  "x": 120,
  "y": 80,
  "width": 64,
  "height": 64,
  "type": "png",
  "transparent": true
}
```

- 预览层根据当前缩放比例显示切图框
- 拖拽时将屏幕坐标换算回原图坐标
- 重新裁切时使用原图坐标，不使用显示坐标

#### 验收标准

- 切图后可以继续移动和缩放
- 调整后的导出资产尺寸正确
- 窗口 resize 后切图框不漂移

---

### 3.6 单个切图资产可选转 SVG

#### 目标

用户可以对某一个已切出的元素尝试转换为 SVG，让简单 icon 或几何图形更容易编辑。

#### 产品边界

SVG 转换不是对整张图默认执行，也不承诺复杂图像高质量可编辑。

适合转 SVG 的内容：

- 纯色 icon
- 线性 icon
- 简单几何图形
- 边框图形
- 简单按钮轮廓

不适合转 SVG 的内容：

- 人物
- 商品图
- 复杂插画
- 渐变背景
- 大面积光影和纹理

#### 需求

- 资产卡片提供 `转 SVG` 操作
- 用户手动选择是否转换
- 转换前给出提示：复杂图片建议保留 PNG
- 转换失败时不影响原 PNG
- SVG 功能在 2.0 可标注为实验能力

#### 技术方案

- P0 先接入简单矢量化流程
- 资产数据中增加 `svgData` 或 `svgUrl`
- PNG 和 SVG 两份资产可以共存
- Figma 回填时优先尝试创建 SVG 节点，失败则回退 PNG

#### 验收标准

- 简单 icon 可以生成 SVG
- 复杂图片失败时有明确提示
- 原 PNG 不会被覆盖或丢失

---

### 3.7 切图后背景修复优化

#### 目标

用户切走某个元素后，原设计稿上的该区域可以弱化或被自然填充，方便看出哪些元素已经处理过。

#### 需求

- 已切图区域可以隐藏或弱化
- 不要出现明显突兀色块
- 支持开关“隐藏已切元素”
- 原始图片必须保留，不可被破坏

#### 技术方案

- 保留 `originalImage`
- 额外生成 `patchedPreviewImage`
- P0 使用周边颜色采样 + 模糊 + 轻微噪声填充
- P1 再考虑更高级的图像修复

#### 验收标准

- 已切区域不会继续强烈干扰视觉
- 填充效果比纯色块自然
- 可以随时切回原图查看

---

### 3.8 透明 PNG 资产输出

#### 目标

切出的资产可以输出透明底 PNG，并保留位置数据。

#### 需求

- 单个资产透明化
- 批量透明化
- 显示透明状态
- 保留原始裁切图
- 保留透明图
- 透明处理失败时可回退原图

#### 技术方案

- Canvas 基于原图坐标裁切
- 简单背景移除或 alpha mask
- 资产数据记录：

```json
{
  "name": "icon_calendar",
  "x": 120,
  "y": 80,
  "width": 64,
  "height": 64,
  "pngDataUrl": "...",
  "transparentPngDataUrl": "..."
}
```

#### 验收标准

- 导出 PNG 背景透明
- 图片边缘没有明显白边或灰边
- 坐标和尺寸准确

---

### 3.9 资产面板升级

#### 目标

切图资产需要有专业管理区，而不是只散落在画布下面。

#### 需求

- 资产缩略图
- 资产名称
- 尺寸
- 类型：PNG / SVG
- 透明状态
- 操作：重命名、删除、透明化、转 SVG、导入 Figma
- 支持批量导出

#### 技术方案

- 使用统一资产状态 `slices[]`
- 所有资产操作只更新 `slices[]`
- UI 基于 `slices[]` 重新渲染
- 后续批量导出和 Figma 回填也读取同一份数据

#### 验收标准

- 用户可以清楚看到当前有多少资产
- 每个资产状态清楚
- 删除和重命名不会影响其他资产

---

### 3.10 Figma 回填优化

#### 目标

点击 `放入 Figma` 后，Figma 画布结构更清晰，更适合后续设计整理。

#### 需求

- 自动创建原始设计稿 Frame
- 自动创建切图资产 Frame
- 自动创建坐标复原 Frame
- 资产按原始坐标放回对应位置
- PNG / SVG 资产区分展示

#### 技术方案

- `ui.html` 发送当前设计稿和 `slices[]` 到 `code.js`
- `code.js` 创建：
  - `Original Screen`
  - `Sliced Assets`
  - `Positioned Assets`
- 使用图片真实尺寸创建 Figma Frame
- 资产节点按 `x/y/width/height` 定位

#### 验收标准

- Figma 中结构清晰
- 原图和资产不会混乱堆在一起
- 坐标复原版本和插件预览一致

## 4. P1 范围

P1 是 P0 稳定后的增强能力。

- 键盘微调切图框
- 批量重命名规则
- 自动类型识别：icon、button、banner、card
- 自动切图建议
- 更自然的背景修复
- 切图资产 zip 导出
- 设计质量检查
- 生成前提示词优化建议

## 5. P2 范围

P2 是后期探索能力，不进入 2.0 主线。

- 整张设计稿结构化还原为 Figma 可编辑图层
- 更强的 SVG 编辑化
- 自动识别文本并重建文字图层
- 自动生成设计系统 token
- 基于 design-md 或类似规范生成可编辑设计稿
- Sketch 输出暂不考虑

## 6. 推荐实施顺序

### 第一轮：窗口和配置基础

1. 插件窗口 resize
2. 迷你折叠模式
3. 响应式布局
4. API 配置持久化

原因：这是后续所有功能的体验底座。

### 第二轮：多图和切图编辑

1. 多设计稿选择流程
2. 切图区域可拖拽
3. 切图区域可缩放
4. 切图重命名和删除

原因：这是 2.0 的核心操作体验。

### 第三轮：资产管理和 Figma 回填

1. 资产面板升级
2. 透明 PNG 状态管理
3. Figma 回填结构优化

原因：让切出来的东西真正可管理、可交付。

### 第四轮：背景修复和 SVG 实验

1. 切图后背景弱化/修复
2. 单个资产转 SVG
3. 失败回退和提示

原因：这两项都有不确定性，适合在核心流程稳定后增强。

## 7. 关键技术风险

### 6.1 SVG 转换质量不可控

复杂图像不适合 SVG。产品上必须明确这是“单个资产手动触发的实验能力”，而不是默认能力。

### 6.2 背景修复可能不自然

P0 只做视觉上不突兀的遮盖，不承诺专业级图像修复。

### 6.3 窗口 resize 后坐标漂移

切图坐标必须永远基于原图真实尺寸。显示层可以缩放，但数据层不能用显示尺寸。

### 6.4 API Key 保存安全

配置只能保存在本地，不能写进源码，也不能提交到仓库。

### 6.5 多图状态复杂

每张生成图必须有独立 `imageId`。切图资产必须绑定图片 ID，避免切换图片后资产错乱。

## 8. 数据结构草案

### 8.1 生成图

```json
{
  "id": "image_001",
  "url": "data:image/png;base64,...",
  "width": 750,
  "height": 1624,
  "prompt": "生成一个健身 App 首页",
  "provider": "openrouter",
  "model": "gpt-5.4-image-2",
  "createdAt": "2026-05-06T10:00:00+08:00"
}
```

### 8.2 切图资产

```json
{
  "id": "slice_001",
  "imageId": "image_001",
  "name": "icon_calendar",
  "x": 620,
  "y": 120,
  "width": 64,
  "height": 64,
  "type": "png",
  "transparent": true,
  "pngDataUrl": "data:image/png;base64,...",
  "transparentPngDataUrl": "data:image/png;base64,...",
  "svgData": null
}
```

### 8.3 用户配置

```json
{
  "provider": "openrouter",
  "baseUrl": "https://openrouter.ai/api/v1",
  "model": "gpt-5.4-image-2",
  "apiKeySaved": true,
  "window": {
    "width": 1440,
    "height": 900,
    "collapsed": false
  }
}
```

## 9. 2.0 验收口径

2.0 完成时，至少需要满足：

1. 用户可以调整插件窗口大小，并且布局不乱
2. 用户保存一次 API 配置后，下次打开自动读取
3. 用户生成多张图后，可以选择其中一张进入切图
4. 用户切图后可以调整区域大小和位置
5. 用户可以管理切图资产
6. 用户可以把资产放回 Figma，结构清晰
7. 用户可以对单个资产尝试转 SVG
8. 用户可以对单个切图资产触发 AI 重绘 SVG，得到更干净的可编辑资产
9. 敏感配置不会进入 GitHub
