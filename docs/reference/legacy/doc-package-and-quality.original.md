## 6. v0.1 的核心产品结论

v0.1 只验证一件事：

```text
用户上传 PNG 后，系统能否快速生成接近原图的 Figma 可编辑设计稿。
```

判断 v0.1 是否成功，不看功能数量，而看：

```text
生成得出来
看起来接近
主要内容能编辑
速度可接受
失败能定位
```

------

## 7. v0.1 的核心技术结论

v0.1 的技术核心是两件事：

```text
1. DSL v0.1
2. Image-to-Figma Renderer
```

DSL 是后端与 Figma 插件之间的数据协议。

Renderer 是把 DSL 转成 Figma 可编辑图层的核心渲染包。

核心链路：

```text
PNG
↓
后端识别
↓
DSL v0.1
↓
Renderer
↓
Figma 可编辑稿
```

------

## 8. v0.1 的开发原则

开发过程中必须遵守以下原则：

```text
1. 先跑通主链路
2. 先让 DSL → Figma 跑通
3. 再让 PNG → DSL 跑通
4. 不为 P1 / P2 功能打断 MVP
5. 不为了 100% 可编辑拖慢生成
6. 复杂区域允许 fallback
7. 不做多轮 AI 复杂修复
8. 不做插件内质量报告
9. 不做插件内对比工具
10. 不做商业化系统
```

------

## 9. v0.1 的开发重点

v0.1 必须优先完成：

```text
DSL Schema
示例 DSL
Renderer 渲染 root / text / shape / image
Figma 插件最小 UI
后端 upload / task / dsl API
SQLite 数据模型
本地文件存储
OCR 接入
AI 结构分析
DSL Builder
图片资产裁切
完整链路联调
```

## 10. v0.1 的非目标

v0.1 不追求：

```text
100% 像素级一致
100% 可编辑
100% 图标矢量化
100% 组件化
100% Auto Layout

100% 商业化完整
```

```text

复杂区域图片兜底
整体视觉接近

链路稳定
```

# 速度与质量取舍原则 v0.1

文档名称：速度与质量取舍原则

## 1. 文档目的

本文档用于定义 Image-to-Figma Design v0.1 在生成速度、视觉还原、可编辑性和系统复杂度之间的取舍原则。

本项目的核心链路是：

```text
PNG 上传
→ 后端识别
→ 生成 DSL v0.1
→ Figma 插件渲染
→ Figma 可编辑设计稿
```

v0.1 的目标不是做一个功能最全的系统，而是做一个能快速验证主链路的 MVP。

因此必须防止开发过程中出现以下问题：

```text
为了更可编辑，生成速度变得很慢
为了更像，节点数量爆炸
为了更智能，AI 调用变成多轮复杂流程
为了更完整，MVP 被 P1 / P2 功能拖死
```

------

## 2. v0.1 的最高原则

v0.1 的最高原则是：

```text
先生成得出来，再追求生成得更好。
```

更具体地说：

```text
快
像
主要内容可编辑
复杂区域兜底
失败可追踪
```

不是：

```text
全可编辑
全组件化
全 Auto Layout
全矢量化
全自动修复
全像素级一致
```

------

## 3. 四个核心目标的优先级

v0.1 的目标优先级如下：

```text
1. 稳定生成
2. 视觉接近
3. 主要内容可编辑
4. 生成速度可接受
```

其中“稳定生成”优先级最高。

如果某个复杂区域会导致整页失败，应优先 fallback。

如果某个高级能力会导致生成明显变慢，应后置。

------

## 4. 速度目标

v0.1 建议内部速度目标：

```text
简单移动端页面：15～30 秒
中等复杂页面：30～60 秒
复杂页面：60～90 秒
超复杂页面：允许失败或 fallback
```

超过：

```text
120 秒
```

应返回超时或失败提示，不应无限等待。

------

## 5. AI 调用取舍原则

### 5.1 v0.1 推荐策略

普通页面：

```text
OCR / CV 预处理
→ 1 次主 AI 结构分析
→ 生成 DSL
```

异常情况：

```text
主 AI 输出 JSON 异常
→ 最多 1 次 JSON repair
```

### 5.2 v0.1 不推荐策略

不要做：

```text
页面类型分析一轮
布局分析一轮
文本分析一轮
组件分析一轮
图标分析一轮
图片分析一轮
质量评分一轮
低分修复一轮
修复后再评分一轮
```

这种多轮链路会显著拖慢生成速度。

### 5.3 结论

v0.1 推荐：

```text
普通页面最多 1 次主 AI 调用
异常 JSON 最多再 1 次 repair
```

不做复杂多轮模型链路。

------

## 6. OCR 与 AI 的分工

OCR 负责：

```text
文字识别
文字 bbox
文字置信度
文本块合并
```

AI 负责：

```text
结构理解
元素归属
组件 role 判断
复杂区域 fallback 判断
DSL 结构辅助生成
```

AI 不应该自由改写所有 OCR 文本。

尤其以下文本必须保守：

```text
价格
手机号
订单号
日期
百分比
编号
金额
库存
```

对这类文本，策略是：

```text
OCR 优先
AI 只做低风险纠错
不做自由改写
```

------

## 7. 可编辑性与速度的取舍

v0.1 不追求 100% 可编辑。

应该可编辑的内容：

```text
标题文字
按钮文字
列表文字
价格文字
表单文字
TabBar 文字
普通说明文字
按钮背景
卡片背景
基础分割线
基础图标
```

可以不编辑、直接图片兜底的内容：

```text
复杂 Banner
复杂插图
复杂运营图
复杂图表
复杂 Logo
复杂多色图标
复杂渐变背景
状态栏
复杂光效
复杂装饰
```

核心原则：

```text
主要内容可编辑，复杂内容图片兜底。
```

------

## 8. 视觉还原与可编辑性的取舍

如果“可编辑”会明显破坏视觉还原，优先保证视觉接近。

例如：

```text
复杂 Banner 拆成多个图层后严重错位
→ 直接 fallback 为图片

复杂图表拆成 shape 后不准确
→ 直接 fallback 为图片

复杂多色图标无法准确矢量化
→ 直接 fallback 为图片
```

原因：

```text
用户第一眼感知的是像不像。
主要内容能编辑即可。
复杂区域不必强拆。
```

------

## 9. 节点数量取舍原则

Figma 节点过多会导致：

```text
渲染慢
插件卡
画布操作卡
图层面板混乱
用户难以编辑
```

v0.1 建议单页节点数：

```text
简单页面：50～150
中等页面：100～300
复杂页面：不建议超过 500
```

如果一个复杂区域会拆出大量节点，应该 fallback。

例如：

```text
复杂插画可能拆出 300 个 Vector
→ 不拆，直接图片

复杂 Banner 可能拆出 100 个 Shape
→ 不拆，直接图片

复杂图表可能拆出 200 个线条和文字
→ v0.1 不结构化，直接图片
```

------

## 10. 图片资产数量取舍原则

图片资产过多会导致：

```text
后端裁切慢
上传 / 访问慢
插件 fetch 慢
Figma createImage 慢
错误率增加
```

v0.1 建议裁切图片资产：

```text
原图参考层
头像
商品图
Banner
Logo
复杂 fallback 区域
必要的图标 fallback
```

不要裁切：

```text
大量小碎图
每一个小装饰
每一个阴影
每一个复杂纹理
```

建议单页图片资产数量：

```text
普通页面：5～20
复杂页面：不建议超过 30
```

------

## 11. Fallback 原则

Fallback 是 v0.1 的核心策略，不是失败。

需要 fallback 的情况：

```text
复杂区域识别置信度低
结构化后视觉会变差
节点数量会爆炸
图片区域可编辑价值低
复杂图表难以准确还原
复杂 Banner 拆解成本高
```

Fallback 应记录：

```text
elementId
assetId
reason
sourceBBox
confidence
```

Fallback 的目标：

```text
保证整体视觉完整
保证主链路不崩
保证生成速度可控
保证后续可定位优化
```

------

## 12. 自动修复取舍原则

v0.1 只做轻量自动修复。

允许修复：

```text
缺 role → unknown
缺 name → 自动补
缺 style → {}
缺 children → []
缺 opacity → 1
缺 visible → true
坐标小数归一
非法宽高直接跳过或 fallback
asset 缺失记录错误
```

不做：

```text
低分页面多轮修复
自动评分后重新生成
自动回滚
多轮 AI repair
复杂布局重新推断
复杂组件重新归组
```

原因：

```text
自动修复闭环会增加耗时、成本和复杂度。
v0.1 先保证一次生成可用。
```

------

## 13. 评分系统取舍原则

v0.1 不把质量评分放进用户主链路。

不做：

```text
生成后自动截图
截图与原图像素对比
输出质量分
低分自动修复
修复后重新评分
插件内展示还原度
```

原因：

```text
评分系统会拖慢用户生成链路。
v0.1 先把生成链路跑通。
```

可以在内部测试中离线做：

```text
人工评审
样例对比
错误分类
低分样本记录
```

------

## 14. 插件 UI 取舍原则

v0.1 插件 UI 只做：

```text
上传
预览
生成中
完成
失败
```

不做：

```text
历史记录
质量报告
插件内对比
设置页
模型选择
高级参数
调试面板
批量任务列表
```

原因：

```text
插件 UI 不是 v0.1 最大风险。
v0.1 最大风险是 PNG → DSL → Figma 是否能跑通。
```

------

## 15. Renderer 取舍原则

Renderer 必须简单、稳定、快速。

Renderer 只做：

```text
DSL → Figma 图层
```

Renderer 不做：

```text
OCR
AI
DSL 生成
图片裁切
质量评分
组件化
Auto Layout
业务语义判断
```

Renderer 遇到局部错误：

```text
单个元素失败 → 跳过并记录 warning
图片加载失败 → 占位或跳过
字体失败 → 默认字体
icon 找不到 → 跳过或占位
root 失败 → 整体失败
```

------

## 16. 后端 Pipeline 取舍原则

v0.1 后端 Pipeline 应保持线性：

```text
upload
→ preprocess
→ ocr
→ ai_analyze
→ asset_crop
→ dsl_build
→ dsl_validate
→ completed
```

不做复杂分支：

```text
多模型并行
多轮修复
评分后重跑
多策略融合
复杂缓存
复杂任务编排
```

如果识别失败：

```text
记录错误
返回失败
不要无限重试
```

------

## 17. 页面类型取舍原则

v0.1 优先级：

```text
P0：App / 小程序移动端页面
P1：后台系统基础兼容
P2：Web / Landing Page 暂不重点支持
```

移动端页面重点支持：

```text
顶部导航
状态栏
搜索框
TabBar
按钮
卡片
列表
表单
弹窗
详情页
长页面
```

后台系统只做基础兼容：

```text
侧边栏
顶部栏
筛选区
表格
卡片
按钮
弹窗
分页
```

复杂后台图表和复杂表格允许 fallback。

## 21. 质量判断优先级

当开发过程中出现冲突时，按以下优先级判断：

```text
1. 是否能稳定生成
2. 是否整体视觉接近
3. 主要文字是否可编辑
4. 主要图片是否正确
5. 是否速度可接受
6. 图层结构是否清晰
7. 是否更高级可编辑
```

不要为了第 7 点牺牲前 5 点。

------

## 22. 什么时候应该 fallback

以下情况应直接 fallback：

```text
拆解后视觉明显变差
拆解后节点太多
识别置信度低
区域本身可编辑价值低
模型输出不稳定
Renderer 实现成本高
会显著拖慢生成
```

具体示例：

```text
运营 Banner
复杂插图
多色 Logo
复杂图表
玻璃拟态背景
复杂渐变光效
装饰性纹理
状态栏
```

------

## 23. 什么时候应该结构化

以下情况应尽量结构化：

```text
导航标题
列表文字
商品标题
价格
按钮文字
输入框
TabBar 文字
卡片背景
按钮背景
普通分割线
常见单色图标
```

因为这些内容：

```text
用户常改
识别相对稳定
可编辑价值高
结构简单
```

------

## 24. 速度变慢时的优化顺序

如果生成速度变慢，按以下顺序优化：

```text
1. 减少 AI 调用轮次
2. 减少节点数量
3. 增加复杂区域 fallback
4. 减少图片裁切数量
5. 减少图片加载数量
6. 优化 Renderer 图片缓存
7. 优化 OCR / AI prompt
8. 后续再考虑缓存系统
```

不要第一时间做复杂缓存。

------

## 25. 效果变差时的优化顺序

如果生成效果差，按以下顺序优化：

```text
1. 检查 DSL 是否表达正确
2. 检查 Renderer 是否按 DSL 渲染正确
3. 检查图片资产裁切是否正确
4. 检查 OCR 是否识别正确
5. 检查 AI 结构归属是否合理
6. 检查是否应该 fallback
7. 再考虑 prompt 优化
8. 最后再考虑多模型方案
```

不要一上来就堆模型。

------

## 26. v0.1 明确禁止的过度优化

v0.1 不应投入以下方向：

```text
多模型投票
多轮自动修复
复杂质量评分闭环
复杂缓存系统
复杂图标库
复杂 Auto Layout
复杂组件化
完整代码生成
复杂商业化系统
插件内测试报告
```

这些都可能让 MVP 变慢、变重、难上线。

------

## 27. MVP 成功判断

v0.1 成功不是因为功能多，而是因为：

```text
单张 PNG 能成功生成 Figma 可编辑稿
简单页面速度可接受
主要文字可编辑
主要图片位置正确
复杂区域能 fallback
失败有日志可查
```

如果这几点成立，v0.1 就成功。

------

## 28. 版本结论

v0.1 的取舍原则可以总结为：

```text
能编辑的编辑。
复杂的 fallback。
别多轮。
别过度拆。
别拖慢。
先跑通。
```

更完整地说：

> 第一版不要追求完美重建，而要追求快速、稳定、视觉接近、主要内容可编辑。

所有开发决策都要服务：

```text
PNG → DSL → Figma 可编辑稿
```

如果某个功能不能直接提升这条链路，默认后置。

