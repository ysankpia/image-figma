~~~markdown
## 1. 文档目的

本文档用于明确 Image-to-Figma Design 项目第一版 MVP 中不做的功能范围。

第一版只验证核心链路：

```text
PNG 上传
→ 后端识别
→ 生成 DSL v0.1
→ Figma 插件渲染
→ Figma 可编辑设计稿
~~~

凡是不能直接提升这条核心链路可用性的功能，默认不进入 v0.1。

本文档的核心作用是：

```text
防止 MVP 过度膨胀
防止开发范围失控
防止第一版变慢、变重、变复杂
防止还没验证效果就开始做外围功能
```

------

## 2. 一期不做的总原则

一期不做事项遵循以下原则：

```text
1. 不做和 PNG → Figma 主链路无关的功能
2. 不做需要大量 UI / 交互设计但不影响生成效果的功能
3. 不做复杂商业化系统
4. 不做复杂权限和账号体系
5. 不做会显著拖慢生成速度的高级能力
6. 不做需要长期数据验证后才能做好的优化功能
7. 不做会导致开发难度成倍增加的“完美化”功能
```

第一版只追求：

```text
快
像
可编辑
稳定生成
```

------

## 3. 不做代码生成

一期不做任何代码生成能力。

不做：

```text
Figma → HTML
Figma → Vue
Figma → React
Figma → 小程序代码
Figma → Tailwind
Figma → CSS
DSL → 前端代码
```

原因：

```text
第一版的目标是 PNG → Figma 可编辑设计稿，不是设计稿转代码。
代码生成会引入布局语义、响应式、组件复用、样式规范等大量复杂问题。
在 PNG → Figma 还没稳定之前，不应该引入代码生成。
```

后续版本可以考虑：

```text
DSL 稳定后
图层结构稳定后
组件语义稳定后
再讨论代码生成
```

------

## 4. 不做真正 Figma Component / Instance

一期不生成真正的 Figma Component / Instance。

不做：

```text
createComponent
createComponentFromNode
自动创建主组件
自动创建组件实例
自动创建 Variants
自动整理组件库
```

一期只做：

```text
普通 Frame / Group
通过 role 保留组件语义
通过 name 清晰命名
通过 componentSpec 记录后续可组件化信息
```

例如：

```json
{
  "type": "frame",
  "role": "button",
  "name": "Primary Button",
  "meta": {
    "componentSpec": {
      "kind": "Button",
      "variant": "primary",
      "confidence": 0.88
    }
  }
}
```

原因：

```text
输入是 PNG，系统只能推断组件关系。
如果强行生成 Component / Instance，容易把相似但不同的元素错误绑定。
用户后续修改一个实例可能影响不该影响的其他区域。
```

后续增强：

```text
一键组件化 / Convert to Components
在主链路稳定后再做
由用户确认后再执行
```

------

## 5. 不做一键组件化

一期不做：

```text
自动识别重复组件
自动把 Card / Button / TabBar 转成组件
自动创建组件库
自动建立跨页面实例关系
```

原因：

```text
一键组件化依赖大量稳定识别结果。
需要先证明 PNG → DSL → Figma 的主链路可用。
如果主链路还不稳定，组件化会放大错误。
```

后续版本可以做：

```text
识别重复结构
生成候选组件
用户确认
再转 Figma Component
```

## 6. 不做 Auto Layout 自动推断

一期不做 Figma Auto Layout。

不做：

```text
自动判断横向 Auto Layout
自动判断纵向 Auto Layout
自动设置 padding
自动设置 gap
自动设置 hug / fill
自动设置 constraints
自动推断响应式布局
```

一期统一采用：

```text
绝对定位
x / y / width / height
```

原因：

```text
截图还原阶段，绝对定位最稳定。
Auto Layout 需要理解设计意图，不能只靠截图可靠推断。
错误的 Auto Layout 会导致视觉结果偏移，增加调试成本。
```

后续版本可以考虑：

```text
高置信度区域再转 Auto Layout
用户手动触发优化
只对列表、按钮、TabBar 等稳定结构做局部 Auto Layout
```

## 7. 不做插件内质量分

一期插件 UI 不展示质量分。

不做：

```text
还原度 86%
OCR 得分
布局得分
样式得分
可编辑率
fallback 比例
错误区域数量
```

原因：

```text
质量分更适合内部测试和模型优化。
普通用户第一版只需要知道生成是否完成。
如果过早展示质量分，会增加解释成本，也会给用户造成困惑。
```

一期只展示：

```text
上传中
处理中
生成中
生成完成
生成失败
```

质量评估放到内部测试后台或离线测试流程。

第一版不做：

