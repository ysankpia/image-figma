# Definition of Done

一个任务完成必须同时满足实现、验证、文档三件事。

## Documentation Work

文档任务完成条件：

- 文档放在正确目录。
- `docs/index.md` 可导航到它。
- 相对链接可用。
- 没有模板占位符残留。
- 不和现有正式文档冲突。
- 如果替代旧草稿，旧草稿在 legacy 中可追踪。

## Implementation Work

后续实现任务完成条件：

- 代码实现符合当前计划。
- 相关 DSL、API、数据模型或用户流程文档已更新。
- 测试或验证证据已记录。
- 没有把 P1/P2 功能塞进 P0。
- 没有打破模块边界。

## 阶段提交纪律

M 阶段工作必须按阶段提交，不能把多个阶段塞进一个提交。

阶段完成顺序固定为：

1. 完成本阶段计划内代码、测试、文档、ADR 和计划更新。
2. 确认 staged 内容只属于当前阶段，不包含下一阶段探索、临时调试、`backend/storage/`、插件 `dist/`、密钥或无关本地改动。
3. 创建独立 git commit。提交信息应描述阶段能力，而不是笼统写 `update` 或 `misc`。
4. 在该提交之上运行阶段要求的验证命令。
5. 验证通过后才能开始下一阶段；验证失败时，用同阶段 fix commit 修正并重新验证。

如果用户本地已有无关未提交改动，必须先识别并隔离；除非用户明确要求，不得把它们并入阶段提交。

## Bug Work

Bug 修复完成条件：

- 有 bug 记录。
- 有复现说明。
- 有根因。
- 有修复说明。
- 有回归保护。
- 有验证证据。

## MVP Work

MVP 阶段完成条件：

- 主链路可跑通。
- P0 验收通过。
- 样例验收记录完整。
- 失败能定位阶段和错误码。
- 下一阶段遗留问题记录清楚。
- 当前阶段已有独立 commit，下一阶段开始前工作区干净或无关改动已明确隔离。
